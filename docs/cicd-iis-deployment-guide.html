CI/CD Deployment Guide

Step-by-step guide to set up GitHub Actions CI/CD pipeline for deploying an ASP.NET Core 6 web application to IIS on Windows Server.

Windows Server 2025 IIS 10 .NET 6 GitHub Actions Self-Hosted Runner

Architecture Overview

DeveloperPush to main branch
→
GitHubTriggers workflow
→
Self-Hosted RunnerOn the target VM
→
IIS SiteLive on port 8080

The self-hosted runner runs on the same Windows Server VM as IIS. When a developer pushes to main, GitHub notifies the runner, which then builds, tests, and deploys the application directly to the IIS site folder — no external access or deploy keys required.

Step 1 — Enable IIS Features

1

Install IIS and ASP.NET 4.8 Windows Features

Run the following in an elevated PowerShell window. This installs IIS along with the ASP.NET 4.8 and .NET extensibility features required to host ASP.NET Core apps.

# Install IIS + ASP.NET 4.8 features
Install-WindowsFeature `
    -Name Web-Server, Web-Asp-Net45, Web-Net-Ext45, `
           NET-Framework-45-ASPNET `
    -IncludeAllSubFeature `
    -IncludeManagementTools

Verify IIS is installed and running:

Get-Service W3SVC | Select-Object Status, Name, DisplayName
✅
Expected output

Status: Running, Name: W3SVC, DisplayName: World Wide Web Publishing Service

Step 2 — Install .NET 6 SDK

2

Install .NET 6 SDK via winget

The SDK is required on the build server (the same VM) so the runner can compile the project.

winget install Microsoft.DotNet.SDK.6 `
    --accept-source-agreements `
    --accept-package-agreements `
    --silent

Verify the installation:

& "C:\Program Files\dotnet\dotnet.exe" --version
# Expected: 6.0.xxx
ℹ️
Note on .NET version

If your project targets a different .NET version (e.g. net7.0, net8.0), replace Microsoft.DotNet.SDK.6 with Microsoft.DotNet.SDK.7 or Microsoft.DotNet.SDK.8 accordingly.

Step 3 — Install ASP.NET Core Hosting Bundle

3

Install the .NET 6 Hosting Bundle

The Hosting Bundle installs the .NET runtime and the ASP.NET Core Module (ANCM) — the IIS module that forwards HTTP requests to the Kestrel server inside the .NET process.

winget install Microsoft.DotNet.HostingBundle.6 `
    --accept-source-agreements `
    --accept-package-agreements `
    --silent

After installing, restart IIS to load the ANCM module:

Stop-Service W3SVC -Force
Start-Service W3SVC
Write-Host "IIS restarted"
⚠️
Hosting Bundle restart is mandatory

IIS must be restarted after installing the Hosting Bundle, otherwise ASP.NET Core apps will return HTTP 500.19 errors.

Step 4 — Create the IIS Site

4

Create deploy folder, App Pool, and IIS Website

Replace the variables at the top with values appropriate for your server.

# ── Variables — change these for each server ──────────────
$siteName    = "SampleWebApp"        # IIS site name
$appPool     = "SampleWebAppPool"    # IIS application pool name
$sitePath    = "C:\inetpub\SampleWebApp"  # Physical path for site files
$port        = 8080                  # HTTP port for this site
# ──────────────────────────────────────────────────────────

Import-Module WebAdministration

# 1. Create the deploy directory
New-Item -ItemType Directory -Path $sitePath -Force | Out-Null

# 2. Create Application Pool (no managed runtime = ASP.NET Core mode)
if (-not (Test-Path "IIS:\AppPools\$appPool")) {
    New-WebAppPool -Name $appPool
    Set-ItemProperty "IIS:\AppPools\$appPool" `
        -Name managedRuntimeVersion -Value ""
    Write-Host "App pool created: $appPool"
}

# 3. Create the IIS Website
if (-not (Test-Path "IIS:\Sites\$siteName")) {
    New-Website -Name $siteName `
                -Port $port `
                -PhysicalPath $sitePath `
                -ApplicationPool $appPool
    Write-Host "IIS site created: $siteName on port $port"
}

# 4. Verify
Get-WebSite -Name $siteName | Select-Object Name, State, PhysicalPath
ℹ️
Why managedRuntimeVersion = empty?

ASP.NET Core apps run their own runtime (Kestrel). Setting the App Pool to "No Managed Code" prevents IIS from trying to load the old CLR, which would conflict.

Step 5 — Open Windows Firewall Port

5

Add an inbound firewall rule for the site port

Replace 8080 with your chosen port number.

$port     = 8080
$ruleName = "IIS SampleWebApp Port $port"

New-NetFirewallRule `
    -DisplayName $ruleName `
    -Direction   Inbound `
    -Protocol    TCP `
    -LocalPort   $port `
    -Action      Allow `
    -Profile     Any

Write-Host "Firewall rule added: $ruleName"
⚠️
Cloud / Hosted VMs — check your provider's firewall too

If the VM is hosted on Azure, AWS, or a similar provider, you must also open the port in the Network Security Group (NSG) or Security Group from the provider's portal. Windows Firewall alone is not enough.

Step 6 — Register the GitHub Actions Self-Hosted Runner

6a

Get the runner registration token from GitHub

  1. Go to your GitHub repository
  2. Click Settings → Actions → Runners
  3. Click New self-hosted runner
  4. Select Windows as the OS
  5. Copy the token from the --token line in the Configure section
⚠️
Token expires in 1 hour

Use the token immediately after copying. If it expires, generate a new one by repeating the steps above.

6b

Download and extract the runner

# Create runner directory
New-Item -ItemType Directory -Path "C:\actions-runner" -Force | Out-Null

# Download latest runner (check https://github.com/actions/runner/releases for latest version)
$runnerVersion = "2.323.0"
$runnerUrl = "https://github.com/actions/runner/releases/download/v$runnerVersion/actions-runner-win-x64-$runnerVersion.zip"

Invoke-WebRequest -Uri $runnerUrl -OutFile "C:\actions-runner\runner.zip"

# Extract
Add-Type -AssemblyName System.IO.Compression.FileSystem
[System.IO.Compression.ZipFile]::ExtractToDirectory(
    "C:\actions-runner\runner.zip",
    "C:\actions-runner"
)
Remove-Item "C:\actions-runner\runner.zip"
Write-Host "Runner extracted successfully"
6c

Configure (register) the runner

Replace the values in angle brackets with your actual details.

Set-Location "C:\actions-runner"

.\config.cmd `
    --url   "https://github.com/<OWNER>/<REPO>" `
    --token "<PASTE_TOKEN_HERE>" `
    --name  "windows-iis-runner" `
    --labels "self-hosted,Windows,iis" `
    --runnergroup Default `
    --work  "_work" `
    --unattended
✅
Expected output

Runner successfully added and Settings Saved.

Step 7 — Run the Runner as a Windows Service

7

Create a Scheduled Task so the runner starts automatically

This ensures the runner restarts after a VM reboot without manual intervention.

$action    = New-ScheduledTaskAction `
    -Execute          "C:\actions-runner\run.cmd" `
    -WorkingDirectory "C:\actions-runner"

$trigger   = New-ScheduledTaskTrigger -AtStartup

$settings  = New-ScheduledTaskSettingsSet `
    -RestartCount    3 `
    -RestartInterval (New-TimeSpan -Minutes 1) `
    -ExecutionTimeLimit (New-TimeSpan -Hours 0)

$principal = New-ScheduledTaskPrincipal `
    -UserId    "SYSTEM" `
    -LogonType ServiceAccount `
    -RunLevel  Highest

Register-ScheduledTask `
    -TaskName  "GitHubActionsRunner" `
    -Action    $action `
    -Trigger   $trigger `
    -Settings  $settings `
    -Principal $principal `
    -Force

# Start it immediately
Start-ScheduledTask -TaskName "GitHubActionsRunner"
Start-Sleep -Seconds 3
Get-ScheduledTask -TaskName "GitHubActionsRunner" | Select-Object TaskName, State
✅
Expected output

TaskName: GitHubActionsRunner | State: Running

ℹ️
Verify runner is online

Go to GitHub repo → Settings → Actions → Runners. The runner should show as Idle (green dot) within 30 seconds.

Step 8 — Add the GitHub Actions Workflow File

8

Add git to the system PATH (required for actions/checkout)

The runner process must be able to find git.exe, otherwise actions/checkout falls back to an API-based download that fails for repos with a .github folder.

$gitPath     = "C:\Program Files\Git\cmd"
$currentPath = [System.Environment]::GetEnvironmentVariable("PATH", "Machine")

if ($currentPath -notlike "*$gitPath*") {
    [System.Environment]::SetEnvironmentVariable(
        "PATH", "$currentPath;$gitPath", "Machine"
    )
    Write-Host "Git added to system PATH"
}

# Restart runner to pick up new PATH
Stop-ScheduledTask  -TaskName "GitHubActionsRunner"
Start-Sleep -Seconds 2
Start-ScheduledTask -TaskName "GitHubActionsRunner"
8b

Create the workflow file in your repository

Create the file .github/workflows/deploy-to-iis.yml in your repository with the content below. Adjust the env variables to match your project paths, port, and pool name.

name: Build and Deploy to IIS

on:
  push:
    branches: [ main ]
  workflow_dispatch:      # allows manual trigger from GitHub UI

jobs:
  build-and-deploy:
    runs-on: [self-hosted, Windows, iis]

    env:
      # ── Change these for your project ─────────────────────────
      SOLUTION_PATH: path/to/YourApp.sln
      PROJECT_PATH:  path/to/YourApp/YourApp.csproj
      TEST_PATH:     path/to/YourApp.Tests/YourApp.Tests.csproj
      PUBLISH_DIR:   C:\deploy\publish
      IIS_SITE_PATH: C:\inetpub\YourApp
      APP_POOL:      YourAppPool
      # ──────────────────────────────────────────────────────────

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup .NET 6
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '6.0.x'

      - name: Restore dependencies
        run: dotnet restore ${{ env.SOLUTION_PATH }}

      - name: Build solution
        run: dotnet build ${{ env.SOLUTION_PATH }} --configuration Release --no-restore

      - name: Run tests
        run: dotnet test ${{ env.TEST_PATH }} --configuration Release --no-build --verbosity normal

      - name: Publish
        run: dotnet publish ${{ env.PROJECT_PATH }} --configuration Release --output ${{ env.PUBLISH_DIR }} --no-build

      - name: Stop IIS App Pool
        shell: powershell
        run: |
          Import-Module WebAdministration
          Stop-WebAppPool -Name "${{ env.APP_POOL }}"
          Start-Sleep -Seconds 2

      - name: Deploy to IIS
        shell: powershell
        run: |
          Remove-Item -Path "${{ env.IIS_SITE_PATH }}\*" -Recurse -Force -ErrorAction SilentlyContinue
          Copy-Item  -Path "${{ env.PUBLISH_DIR }}\*"   -Destination "${{ env.IIS_SITE_PATH }}" -Recurse -Force
          Write-Host "Deployed to ${{ env.IIS_SITE_PATH }}"

      - name: Start IIS App Pool
        shell: powershell
        run: |
          Import-Module WebAdministration
          Start-WebAppPool -Name "${{ env.APP_POOL }}"
          Write-Host "Deployment complete"
⚠️
Build the full solution, not just the project

The Restore and Build steps must target the .sln file (not the .csproj). This ensures test projects are also compiled before the test step runs with --no-build.

Step 9 — Trigger the First Deployment

9

Push the workflow file to trigger the first run

# Clone the repo (if not already cloned)
git clone https://github.com/<OWNER>/<REPO>.git
cd <REPO>

# Create the workflow directory
mkdir -p .github/workflows

# Paste the workflow YAML into .github/workflows/deploy-to-iis.yml
# then commit and push

git commit -m "Add CI/CD workflow for IIS deployment"
git push origin main

The push immediately triggers the workflow. Monitor progress at:

https://github.com/<OWNER>/<REPO>/actions

Run this PowerShell script after the pipeline completes:

Troubleshooting

Runner shows as Offline on GitHub

Check if the scheduled task is running:

Get-ScheduledTask -TaskName "GitHubActionsRunner" | Select-Object TaskName, State

If state is not Running, start it:

Start-ScheduledTask -TaskName "GitHubActionsRunner"

Check runner logs for errors:

Get-ChildItem "C:\actions-runner\_diag\" -Filter "*.log" |
    Sort-Object LastWriteTime -Descending |
    Select-Object -First 1 |
    ForEach-Object { Get-Content $_.FullName -Tail 30 }

Checkout step fails with "cannot find path .github"

Git is not in the runner's PATH. Fix:

$gitPath = "C:\Program Files\Git\cmd"
$cur     = [System.Environment]::GetEnvironmentVariable("PATH", "Machine")
[System.Environment]::SetEnvironmentVariable("PATH", "$cur;$gitPath", "Machine")
Stop-ScheduledTask  -TaskName "GitHubActionsRunner"
Start-Sleep -Seconds 2
Start-ScheduledTask -TaskName "GitHubActionsRunner"

Test step fails — DLL not found

The Build step is targeting only the web project .csproj, so the test project is never compiled. Fix: change the Restore and Build steps to target the .sln file instead.

Site returns HTTP 500.19 or 500.30

500.19 — Hosting Bundle not installed or IIS not restarted after install.

# Reinstall hosting bundle, then restart IIS
winget install Microsoft.DotNet.HostingBundle.6 --silent
Stop-Service W3SVC -Force; Start-Service W3SVC

500.30 — App failed to start. Check the Windows Event Log:

Get-EventLog -LogName Application -Source "IIS AspNetCore Module*" -Newest 10 |
    Select-Object TimeGenerated, Message | Format-List

Site not reachable from browser (but works on localhost)

Two things to check:

1. Windows Firewall — add inbound rule:

New-NetFirewallRule -DisplayName "IIS Port 8080" -Direction Inbound `
    -Protocol TCP -LocalPort 8080 -Action Allow -Profile Any

2. Cloud NSG / Security Group — open port 8080 inbound from your cloud provider's portal (Azure NSG, AWS Security Group, etc.).
